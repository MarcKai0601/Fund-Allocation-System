# V5 更新紀錄 — RBAC 角色權限控管 & Token 解析強化

**版本**: v5.0.0  
**日期**: 2026-03-06  
**類型**: Security Enhancement — RBAC 通用角色檢查 & Redis Session 雙格式相容

---

## 變更背景

隨著 Java MGR 認證中心逐步升級，Redis Session 資料結構從舊的 PascalCase 格式演進為新的 camelCase 格式。本次升級讓 FAS 後端同時支援新舊兩種格式，並新增通用 RBAC 機制，可在任意 API 端點進行角色檢查。

### Redis Session 格式演進

```
舊格式 (v4.x):  {"UserId": 1, "Roles": {"FAS": ["ADMIN"]}}
新格式 (v5.0):  {"userId": 1, "username": "admin", "roles": [{"roleCode": "ADMIN"}]}
相容模式:       同時包含兩套欄位，新舊程式碼皆可正常運作
Fallback:       純字串 user_id（最舊資料相容）
```

---

## 後端修改

### `app/core/security.py` — 新增 RBAC 元件

| 元件 | 類型 | 說明 |
|------|------|------|
| `UserSession` | Pydantic Model | 統一的 Session 資料結構：`user_id`, `username`, `roles: list[str]` |
| `get_current_user_session` | Dependency | 解析 Session dict → 回傳 `UserSession` 實例（含角色清單） |
| `RequireRole` | Dependency Class | 通用 RBAC 檢查，接收一個或多個角色代碼，不符合拋出 403 |
| `_extract_roles()` | Helper | 將新舊兩種 roles 格式統一為 `list[str]` |

**`get_current_user` 強化**：支援三種 Redis 值格式

```
1. JSON + userId (新 camelCase)    → 正常解析
2. JSON + UserId (舊 PascalCase)   → 正常解析
3. 純字串 (最舊 fallback)            → 視為 user_id 直接回傳
```

**`RequireRole` 使用示範**：

```python
# 單一角色
@router.get("/admin-only")
def admin_endpoint(user: UserSession = Depends(RequireRole("ADMIN"))):
    return {"msg": f"Hello {user.username}"}

# 多角色 (OR 邏輯 — 任一即可)
@router.delete("/resource/{id}")
def delete_resource(user: UserSession = Depends(RequireRole("ADMIN", "MANAGER"))):
    ...
```

### `app/core/redis_client.py` — 雙格式寫入

| 變更 | Before | After |
|------|--------|-------|
| `set_auth_token()` 簽章 | `(token, user_id, roles, ttl)` | `(token, user_id, roles, role_codes, username, ttl)` |
| Token Value | `{"UserId", "Roles"}` | `{"UserId", "Roles", "userId", "username", "roles"}` |

### `app/api/portfolios.py` — 新增管理員端點

| 方法 | 路由 | 權限 | 說明 |
|------|------|------|------|
| GET | `/api/portfolios/all` | `RequireRole("ADMIN")` | **[NEW]** 管理員查看所有使用者的 Portfolios |

既有路由不受影響，仍使用 `require_fas_access` / `get_valid_portfolio`。

### `app/api/dev_auth.py` — 模擬登入擴充

| 變更 | 說明 |
|------|------|
| `username` 欄位 | 新增，預設 `"dev_user"` |
| `role_codes` 欄位 | 新增，預設 `["ADMIN"]`，用於測試新 RBAC 格式 |
| Redis Session | 同時寫入新舊格式，方便測試兩條程式碼路徑 |

## 驗證流程

```
前端 → Authorization: Bearer <token>
         │
         ▼ HTTPBearer 解析
         │
         ▼ Redis GET token:<token>
         │
         ├── 不存在 → 401 Unauthorized
         │
         ▼ 嘗試 JSON 解析
         │
         ├── 非 JSON → fallback: 純字串視為 user_id
         │
         ▼ 提取 userId / UserId
         │
         ├── 都沒有 → 401 Unauthorized
         │
         ▼ get_current_user_session → UserSession(user_id, username, roles)
         │
         ▼ RequireRole("ADMIN") → 比對 roles
         │
         ├── 無交集 → 403 Forbidden
         │
         ▼ 回傳 UserSession → endpoint 操作 ✅
```

## 向後相容性

- ✅ 舊格式 `{"UserId", "Roles"}` 的 Token 仍可正常驗證
- ✅ 純字串 user_id 的 Token 透過 fallback 相容
- ✅ 既有路由（`require_fas_access`, `get_valid_portfolio`）行為不變
- ⚠️ 新 `RequireRole` 僅檢查扁平化的 role codes，需確認 Java 端寫入格式對齊
